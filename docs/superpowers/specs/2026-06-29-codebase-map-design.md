# Codebase Map — mass AI description generation

## Status legend

- **Implemented / Shipped** — merged to the production codebase, available to users.
- **PoC implemented (not delivered)** — prototype, not merged.
- **Planned** — designed here, not yet built.

Everything in this spec is **Planned** unless noted otherwise.

## Goal

Let the user pre-compute AI descriptions for every file in the workspace in one
background pass, with visible progress and pause/resume, so that:

1. Smart-search shows descriptions instantly (served from the persisted index)
   instead of generating them on demand per query.
2. The result is exported as artifacts other agents (e.g. Claude) can read to
   orient themselves in the codebase.

## Background (already in place)

- `WorkspaceIndexer` builds and persists a per-root index; records carry a static
  `description`, an optional `aiDescription`, category, symbols, embeddings.
- `index.setAiDescription(path, text)` mutates a record; `indexer.save()` persists.
- Smart-search already **prefers** `aiDescription`: `resolveOrCollect` in
  `spexr-search-backend-service.ts` checks `record.aiDescription` first, then genuine
  static prose, then the model. So "search uses the precomputed result" needs **no
  change** — it is automatic once records carry `aiDescription`.
- `DescriptionGenerator.generateBatch(items)` runs one path-keyed 0.5B inference for
  a set of files (see decision `2026-06-29-0728-smart-search-ai-descriptions-batch`).

The feature therefore reuses the existing generation + persistence path; the new
work is orchestration, controls/progress UI, and artifact export.

## Decisions (resolved during brainstorming)

| Topic | Decision |
|-------|----------|
| Model | Qwen 0.5B local (reuse `generateBatch`). On-demand path stays 0.5B. |
| Export | Both `.spexr/codebase-map.md` and `.spexr/descriptions.json`. |
| Scope | Default: only files missing `aiDescription`. Separate "Regenerate all" overwrites. |
| Resume | Implicit via `aiDescription` coverage; pause = cooperative cancel between batches. |
| Triggers | CTA `✦ Map this codebase` + ⓘ tooltip in smart-search header **and** Command Palette + status bar. |
| Progress | Determinate (total known), mirrored in widget header and status bar. |
| `.spexr/` | Added to `.gitignore` (regenerable; avoid churn). |
| Save cadence | Persist the index every 5 batches (and on pause/stop/complete). |

## Architecture

### Backend

**`description-job.ts` (new) — per-workspace orchestrator.**
Holds job state and runs the batch loop. Injected with the workspace's `index`, the
`DescriptionGenerator`, a `save()` callback, and a `CodebaseMapWriter`. No direct fs
access except through the writer.

State:
```ts
type DescriptionJobState = "idle" | "running" | "paused" | "complete" | "error";
interface DescriptionJobStatus {
  state: DescriptionJobState;
  done: number;   // targets processed this run (reaches total at completion)
  total: number;  // size of the target set, fixed at start()
  message?: string; // on error
}
```

Behaviour:
- `start({ regenerate })`: ensure the index is ready; compute the target list — all
  records when `regenerate`, else records without `aiDescription`. Fix `total`. Set
  `running`. Drive the batch loop.
- Batch loop: take the next `BATCH_SIZE` targets (= 5, matching the on-demand
  top-N batch size), read their file
  content, call `generateBatch`, `setAiDescription` for non-null results, increment
  `done`. Save every 5 batches. Between batches, honor a pause/stop flag (cooperative
  cancellation — never interrupt an in-flight inference).
- `pause()`: request suspension; loop stops after the current batch, saves, sets
  `paused`.
- `resume()`: continue the loop from remaining targets.
- On exhausting targets: save, write export artifacts, set `complete`.
- Worker crash → `generator.isAvailable()` false → save partial, set `error`.

Files dropped by the model (null from `generateBatch`) stay without `aiDescription`;
they are retried on the next run or via the on-demand path. `done` counts only files
that ended with a description, so progress reflects real coverage.

**`codebase-map-writer.ts` (new) — artifact generation.**
Pure transform from index records → two strings, plus a thin fs write to `<root>/.spexr/`.
- `codebase-map.md`: records grouped by top-level folder, then category, each line
  `relPath — description` (prefers `aiDescription`, falls back to static `description`).
- `descriptions.json`: `{ [relPath]: { description, category } }` where `description`
  is the best available (`aiDescription` if present, else the static `description`).
  Symbols are intentionally omitted: the index does not persist them, and they are
  already present in the source files agents can read.
Both written in a single pass over the index so they never drift.

**`spexr-search-backend-service.ts` — thin delegation.**
Owns one `DescriptionJob` per workspace. New methods forward to it:
`startDescriptionJob(root, { regenerate })`, `pauseDescriptionJob(root)`,
`resumeDescriptionJob(root)`, `getDescriptionJobStatus(root)`. Job progress is pushed
to the client via a new callback.

**`search-protocol.ts` — RPC surface.**
Add the four methods above, `DescriptionJobStatus`, and client callback
`onDescriptionJobProgress(status: DescriptionJobStatus)`.

### Frontend

- **Smart-search widget header:** CTA `✦ Map this codebase` with an adjacent ⓘ info
  icon whose tooltip reads: *"Pre-compute AI descriptions for every file so search is
  instant and agents can orient in the codebase."* While running, the CTA becomes a
  pause/resume toggle with a determinate progress bar (`done/total`). A small overflow
  action offers "Regenerate all".
- **Status bar contribution (`description-job-status-bar-contribution.ts`, new):**
  mirrors job state (`✦ Mapping 142/1500`) with click-to-pause/resume.
- **Commands (Command Palette):** `Spexr: Map this codebase`, `Spexr: Pause mapping`,
  `Spexr: Resume mapping`, `Spexr: Regenerate all descriptions`.

## Data flow

1. User clicks the CTA or runs `Spexr: Map this codebase`.
2. Frontend calls `startDescriptionJob(root, { regenerate: false })`.
3. Backend ensures the index is ready, fixes the target list and `total`, runs the
   batch loop, emits `onDescriptionJobProgress` after each batch.
4. Header and status bar render the determinate progress; pause/resume route to the
   matching RPC calls.
5. On completion the backend writes `.spexr/codebase-map.md` + `descriptions.json`
   and emits `state: "complete"`.
6. Subsequent searches serve `aiDescription` from the index with no change to search
   code; the on-demand path remains a fallback for any file not yet covered.

## Error handling

- Per-file `null` from `generateBatch` → no `aiDescription` written; retried later.
- Generator becomes unavailable (worker crash) → partial progress saved, job `error`,
  message surfaced in header/status bar.
- App restart mid-job → no in-memory job survives, but coverage is persisted, so the
  next `start` resumes from the remaining missing files for free.
- Index not yet built → `start` triggers/awaits the build before looping.

## Testing

- **`description-job` tests** (fake generator/index/writer): target selection
  (missing-only vs regenerate-all), pause stops after the current batch and persists,
  resume continues, progress emission per batch, save cadence (every 5), completion
  triggers the writer exactly once, worker-unavailable → `error` with partial save.
- **`codebase-map-writer` tests** (fake records): MD grouping by folder then category
  and `aiDescription`-over-static preference; JSON shape (`{ description, category }`)
  and key set.
- **RPC wiring:** minimal smoke that backend methods delegate to the job and progress
  reaches a fake client.

## Out of scope (YAGNI)

- Stronger-model (claude CLI) generation — on-demand and mass-gen both stay 0.5B.
- Committing `.spexr/` artifacts (gitignored by default).
- Cross-restart persistence of the *job* object (coverage-based resume covers it).
- Parallel/multi-worker inference.
